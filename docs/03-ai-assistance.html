4  AI 协助过程

4.1 AI 工具选择

本项目在完成过程中使用了 Claude Code 作为主要的 AI 辅助开发工具。选择 Claude Code 的原因包括：

  • 代码理解能力强：能够准确理解现有代码结构和逻辑
  • 多文件协作：可以同时处理多个相关文件，保持一致性
  • Python 和数据科学专长：对 Pandas、Matplotlib、SQL 等数据科技术支持良好
  • 交互式开发：支持边写代码边讨论，及时调整方向 啦啦啦 —

4.2 AI 的主要用途

4.2.1 代码优化与重构

场景 1：移除 tushare 依赖，改用 baostock

原始代码使用 tushare 获取 A 股数据，但 tushare 需要注册且有积分限制。我们希望改用免费的 baostock。

提示词示例： > “请帮我移除 tushare 依赖，改用 baostock 获取 A 股数据。需要获取沪深 300 指数和 10 只代表性股票的日线数据，包括开盘价、最高价、最低价、收盘价、成交量、复权收盘价。”

AI 提供的帮助： - 给出 baostock 的安装和使用方法 - 提供完整的数据获取代码 - 处理数据格式转换，确保与原结构一致 - 添加错误处理和重试机制

4.2.2 图表样式统一与美化

场景 2：统一多个 Notebook 的图表风格

项目中有多个 Notebook，每个都有独立的可视化代码，图表风格不统一。

提示词示例： > “请帮我统一 01_api_download.ipynb 和 03_sql_analysis.ipynb 中的图表风格。要求： > 1. 使用 seaborn 的 whitegrid 风格 > 2. 字体大小统一设置 > 3. 图表尺寸保持一致 > 4. 颜色方案使用专业的配色 > 5. 确保中文能正确显示，如果有问题就用英文标签”

AI 提供的帮助： - 创建统一的样式配置代码块 - 为每个图表应用一致的参数 - 处理中文字体回退机制 - 优化图例位置和标签可读性 - 添加网格线和边框样式调整

效果对比：

优化前 优化后
各图表风格不一 统一专业的视觉风格
字体大小混乱 层级清晰的字体
配色随意 协调的配色方案
中文可能乱码 稳定的显示效果

4.2.3 SQL 查询编写与优化

场景 3：实现复杂的分析查询

需要完成多个 SQL 查询任务，包括必做的 3 个查询和自选的 2 个查询。

提示词示例： > “请帮我实现 03_sql_analysis.ipynb 中的所有查询： > 1. 必做查询 1：计算收益率曲线利差（10Y-2Y），需要用 CASE WHEN 将长表转宽表 > 2. 必做查询 2：计算每只股票的年度平均收盘价和总成交量 > 3. 必做查询 3：筛选银行行业且上市超过 10 年的股票 > 4. 自选查询 1：分析菲利普斯曲线（CPI 与失业率关系） > 5. 自选查询 2：研究股票波动率与成交量的关系 > > 每个查询后都需要有对应的可视化分析和结论解释。”

AI 提供的帮助： - 编写正确的 SQL 查询语句 - 解释每个查询的逻辑和技巧 - 提供可视化代码（Matplotlib/Seaborn） - 帮助解读分析结果，形成有意义的结论 - 补充完整分析流程示例

SQL 查询技巧学习：

在 AI 的帮助下，我们学到了以下实用技巧：

  1. 长表转宽表：使用 CASE WHEN + GROUP BY
  2. 日期处理：使用 substr() 提取年份
  3. 条件筛选：多条件组合使用 AND/OR
  4. 表连接：正确使用 JOIN 关联多个表
  5. 索引利用：AI 提醒我们为常用字段创建索引

4.2.4 文档完善

场景 4：编写项目 README

项目完成后需要一份清晰的 README 文档，方便他人理解和使用。

提示词示例： > “请帮我为这个项目写一份完整的 README.md，包括： > 1. 项目概述 > 2. API 申请说明（FRED 和 baostock） > 3. 项目结构说明 > 4. 配置说明（环境变量和配置文件两种方式） > 5. 数据库结构（每个表的字段说明） > 6. 使用说明（一步步教用户如何运行） > 7. 数据库自动更新说明（包括 Windows 任务计划和 Linux cron） > 8. 注意事项 > 9. AI 工具使用声明”

AI 提供的帮助： - 结构化的文档组织 - 清晰的步骤说明 - 代码示例和配置模板 - 跨平台的操作指南 - 专业的术语解释

4.3 典型的 AI 协作流程

4.3.1 问题-响应-迭代模式

我们与 AI 的协作通常遵循以下流程：

1. 明确任务
   ↓
2. 给出详细提示词
   ↓
3. AI 生成初步方案
   ↓
4. 审查结果，提出修改意见
   ↓
5. AI 根据反馈调整
   ↓
6. （如有需要，重复 4-5）
   ↓
7. 最终确认

4.3.2 示例：图表优化的完整对话

用户： > “这个图的配色不太好看，帮我改得更专业一点。”

AI： > “我来帮你优化图表配色。可以考虑使用 seaborn 的配色方案，比如 ‘viridis’、‘coolwarm’ 或 ‘Set2’。另外，我会调整一下线条粗细和标记样式，让图表更清晰。” > > [提供修改后的代码]

用户： > “配色不错，但 y 轴标签被截断了，能不能调整一下左边距？另外图例放到图的下方吧。”

AI： > “好的，我来调整： > 1. 使用 plt.subplots_adjust(left=0.15) 增加左边距 > 2. 将图例位置改为 loc='upper center', bbox_to_anchor=(0.5, -0.12), ncol=2 放到图下方 > > [提供更新后的代码]”

用户： > “完美，就这样！”

4.4 AI 使用的注意事项

在使用 AI 协助完成作业的过程中，我们也总结了一些经验：

4.4.1 提示词要具体明确

不太好的提示词： > “帮我写个分析代码”

好的提示词： > “帮我写一个 Python 函数，输入是 SQLite 数据库连接，输出是一个 DataFrame，包含每只股票的年化收益率和波动率。使用 2020-2024 年的数据，收益率用复权收盘价计算，波动率用日收益率的标准差乘以 sqrt(252) 年化。”

4.4.2 要审查和验证 AI 的输出

AI 有时会： - 生成看上去正确但实际有逻辑错误的代码 - 使用过时的 API 或已废弃的函数 - 在计算细节上出错（如忘记年化、除权处理等）

我们的做法： - 每段代码都仔细阅读，理解逻辑 - 小范围测试，验证结果合理性 - 交叉核对（比如用两种方法计算同一指标） - 对关键计算，手动抽查几个样本

4.4.3 保持学习心态，而不是直接复制

AI 是学习工具，不是答案机器。我们的目标是：

❌ 不提倡 ✅ 提倡
直接复制粘贴 AI 写的代码 理解 AI 代码的思路和技巧
只问结果，不问过程 让 AI 解释”为什么这么写”
完全依赖 AI 解决问题 用 AI 辅助自己解决问题
不懂装懂 遇到不懂的就问 AI 解释

4.4.4 知道 AI 的边界

AI 很强大，但不是万能的：

  • AI 不擅长：需要真实 API Key 的测试、特定环境配置问题、深度的业务逻辑判断
  • AI 可能出错：复杂的数学计算、对最新库的了解、某些边缘情况的处理
  • AI 没有上下文：它不知道你的具体作业要求、评分标准、团队约定等

应对策略： - 涉及真实 API 调用时，自己测试 - 环境配置问题，多查官方文档 - 重要决策，自己结合作业要求判断

4.5 效率提升对比

使用 AI 前后的工作量对比（估算）：

任务 不使用 AI 使用 AI 节省时间
基础框架搭建 2 小时 30 分钟 ~75%
API 对接 3 小时 1 小时 ~67%
SQL 查询编写 4 小时 1.5 小时 ~62%
可视化美化 3 小时 45 分钟 ~75%
文档编写 2 小时 45 分钟 ~62%
总计 14 小时 4.5 小时 ~68%

更重要的是：AI 帮助我们把精力从”怎么写代码”转移到”怎么分析问题”上，提升了分析的深度和质量。

4.6 本章小结

AI 工具在本项目中的价值总结：

方面 具体体现
效率提升 节省约 68% 的开发时间
代码质量 统一的风格、更好的注释、更健壮的错误处理
学习效果 通过 AI 的解释学到了很多实用技巧
分析深度 有更多精力专注于分析本身，而不是代码实现
文档完善 AI 帮助生成了清晰的 README 和注释

核心经验：AI 是强大的助手，但最终的判断、验证和决策仍需我们自己完成。合理使用 AI，能让我们学得更快、做得更好。
